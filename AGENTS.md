# AGENTS.md

## 基本方針

- 回答は日本語で行う。
- このリポジトリは、TAKAYA UTRシリーズ LAN接続 Pythonサンプルである。
- Codexは、このリポジトリを安全に編集・テストするため、本ファイルの作業ルールに従う。
- このリポジトリは社外公開を前提としたサンプルコードとして扱う。

## 通信仕様の扱い

- 通信仕様は `docs/protocol/` 配下の資料を正とする。
- 通信仕様が不明な場合は、推測で実装しない。
- 不明点は、コードコメント、TODO、確認事項のいずれかとして明記する。
- 仕様書に根拠がない内容を、推測で補完しない。
- 以下の仕様・処理は、根拠なく変更しない。
  - STX / ETX / CR の扱い
  - SUM計算
  - ACK / NACK 判定
  - フレーム長
  - コマンドコード
  - TCP送受信処理
  - Inventory処理
  - ブザー制御処理
  - エラー応答解析
  - タイムアウト処理

## 作業順序

原則として、以下の順序で進める。

1. 現状分析
2. 変更方針の提示
3. ユーザー確認
4. ファイル変更
5. 差分確認
6. pytest または mock テスト
7. ユーザー確認
8. 実機テスト

実機テストは、pytest または mock テストで基本動作を確認してから行う。

## ファイル変更前の確認

ファイルを変更する前に、必ず以下を提示する。

- 変更予定ファイル一覧
- 変更理由
- 影響範囲
- 実機確認が必要な項目
- 変更しない範囲

README、docs、tests、src のいずれを変更する場合も、この確認を行う。

## コード変更時の注意

- 既存の動作仕様を変更しない。
- 通信仕様に関わる処理を変更する場合は、根拠となる仕様書の該当箇所を明示する。
- 大規模リファクタリングは、ユーザーが明示的に許可するまで行わない。
- 既存の1ファイル構成を分割する場合は、先に分割設計案を提示する。
- 仕様が不明な箇所は、仮実装せず TODO として残す。

## テスト方針

- 実機なしで確認できる処理は、pytestで先に確認する。
- 優先してテストする対象は以下とする。
  - SUM計算
  - フレーム生成
  - ACK / NACK 判定
  - NACK解析
  - RSSI変換
  - 受信フレーム解析
  - Inventory応答解析
- 実機通信の前に、可能な範囲で mock サーバーによる確認を行う。
- テストに必要な依存関係は、最小限にする。

## pytest実行結果の扱い

- pytest を実行できなかった場合は、pytest成功として扱わない。
- compileall による構文確認は、pytest の代替とは扱わない。
- compileall 成功は「構文確認成功」としてのみ報告する。
- pytest 未実行の場合は、必ず「pytest未実行」と明記する。
- pytest が実行できた場合のみ、「pytest成功」または「pytest失敗」と報告する。
- Codex環境で `py -m pytest` が実行できない場合は、以下の順で実行を試す。
  1. `python -m pytest`
  2. `where.exe python` で確認した Python 実行ファイルのフルパスを使った pytest 実行
  3. Codex同梱Pythonの有無を確認し、存在する場合はそれを使った pytest 実行
- Codex環境では、OSの `py` / `python` がPATHに存在しない場合がある。
- Codex同梱Pythonの例:
  `C:\Users\tamaru\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe`
- Codex同梱Pythonを使うpytest実行例:
  `& 'C:\Users\tamaru\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe' -m pytest tests -p no:cacheprovider --basetemp "$env:TEMP\utr_lan_pytest_tmp"`
- `logs/real_device/` は実機ログ保存用であり、pytest一時領域には使わない。
- それでも実行できない場合は、理由と実行できなかったコマンドを明記する。
- ユーザー環境でpytest成功が確認済みの場合は、「ユーザー環境ではpytest成功確認済み」と分けて記載する。
- Codex自身が実行していないpytest結果を、自分の確認結果として書かない。

## 実機テスト前の確認

実機テストを行う前に、必ず以下を提示する。

- 対象IP
- ポート
- 対象機種
- 接続方式
- 実行コマンド
- 実行する処理
- 期待結果
- ログ保存先
- 禁止操作

通信先、ポート、コマンド内容が不明な場合は、実機テストを実行しない。

## 実機テストで許可する初期操作

初期の実機テストでは、以下の範囲に限定する。

- TCP接続確認
- ROMバージョン確認
- コマンドモード設定
- Inventory 1回
- ログ保存

以下は、ユーザーが明示的に許可するまで実行しない。

- EEPROM書き換え
- 通信設定変更
- RF出力設定変更
- 初期化
- 連続Inventory
- ファームウェア更新
- ブザー制御

## Git操作

以下の操作は、ユーザーが明示的に許可するまで実行しない。

- `git commit`
- `git push`
- release作成
- PR作成
- ブランチ削除
- force push

## 公開前提の注意

このリポジトリは社外公開前提として扱う。

以下の情報を含めない。

- 顧客名
- 社内情報
- 非公開情報
- 個人情報
- 認証情報
- 実運用環境を特定できる情報
- 実際の社内ネットワーク情報
- 実際の顧客環境のIPアドレス

## Git管理上の問題

README重複など、Git管理上の問題を発見した場合は、すぐに修正せず、まず以下を提示する。

- 問題の原因
- 影響範囲
- 修正案
- 修正後に確認すべきコマンド

## 変更後の報告

ファイルを変更した後は、必ず以下を報告する。

- 変更ファイル一覧
- 主な変更点
- 通信仕様に関わる変更の有無
- 実機確認が必要な項目
- 次に行うべき作業
