## 基本フレーム

コマンドおよびレスポンスは以下の構造を基本とする。

```text
STX / アドレス / コマンド / データ長 / データ部 / ETX / SUM / CR
```

パケット全体の長さは `データ長 + 7` バイト。

> 根拠: 第5章 5.1「コマンド／レスポンスの通信フォーマット」本文p.98
> 根拠: 第5章 5.2「通信フォーマットの詳細」本文p.99

## 実装に必要な主要項目

| 項目 | 要点 |
|---|---|
| STX | `02h`。パケット先頭。 |
| ETX | `03h`。パケット終端。 |
| CR | `0Dh`。改行コード。 |
| データ長 | データ部のバイト数。`00h`から`FFh`。 |
| SUM | STXからETXまでを1バイト単位で加算した下位1バイト。 |
| ACK | コマンド部が `30h`。 |
| NACK | コマンド部が `31h`。 |
| RFタグ読み取りデータ | コマンド部が `6Ch` のレスポンスが返る。 |

> 根拠: 第5章 5.2「通信フォーマットの詳細」本文p.99
> 根拠: 第5章 5.3「SUM の計算方法」本文p.100
> 根拠: 第6章 6.1.4「コマンド逆引き表」本文p.106
> 根拠: 第7章 7.5.1「UHF_Inventory」本文p.263-p.265

## コマンド送受信の基本ルール

- コマンドモードでは、上位機器からのコマンドに対してリーダライタがレスポンスを返す。
- 連続してコマンドを送信する場合は、前コマンドのレスポンス受信後に次コマンドを送信する。
- 一部、レスポンスを返さないコマンドがある。
- コマンドモード以外の自動読み取りモードでは、RFタグ読み取りごとにリーダライタから非同期にレスポンスが返る場合がある。

> 根拠: 第5章 5.4「コマンドレスポンス」本文p.101

## 主要コマンド

| 用途 | コマンド | 詳細コマンド | 備考 |
|---|---|---|---|
| ROMバージョン読み取り | `4Fh` | `90h` | データ部はASCIIで返る。 |

| LED&ブザーの制御 | `4Eh` | `57h` | LEDとブザー同時制御。 |
| エラー情報の読み取り | `4Fh` | `80h` | エラー状態確認。 |

## 初期実機テストの制限

- 初期実機テストでは FLASH 書き込みは禁止。
- 初期実機テストでは RAM 指定のみを対象とする。
- 動作モードを書き込む場合、書き込み対象は `00h: RAM` のみを使用する。
- `10h: FLASH` を指定する操作は、ユーザーが明示的に許可するまで実行しない。
- EEPROM、通信設定、RF出力設定、初期化、連続Inventory、ファームウェア更新、ブザー制御は、ユーザーが明示的に許可するまで実行しない。

> 根拠: 第7章 7.4.16「リーダライタ動作モードの書き込み」本文p.188-p.189

## タイムアウトの扱い

- RFタグ通信時、リーダライタはRFタグからの応答待ちで一定時間後にタイムアウトし、次処理またはNACKレスポンス送信へ移る。
- コマンドタイムアウト時間の初期値は 20ms。
- 設定可能範囲は 0msから20ms。
- ISO18000-63規格上、読み書きコマンドへのRFタグ応答最大時間は 20ms とされる。
- ホスト側Python実装で使用するソケット受信タイムアウト値は、このPDFだけでは確定しない。

> 根拠: 第3章 3.6「コマンドタイムアウト時間」本文p.54-p.55
